CHAPTER 1: THE PROBLEM AND ITS BACKGROUND
================================================================================

Introduction

In today's world, many organizations operate with employees scattered across
different locations. Whether it's maintenance crews, delivery services, construction
teams, or field inspectors, managing attendance for these workers has always been
a challenge. Traditionally, companies relied on paper-based logs or simple check-in
systems that could easily be manipulated. An employee might claim to be at a job
site when they're actually somewhere else, or someone could check in on behalf of
another worker. These problems have plagued organizations for years, leading to
inaccurate records and wasted resources.

In recent years, digital solutions have started to emerge. However, many of these
systems come with significant drawbacks. They often depend on expensive third-party
services, require ongoing subscription fees, or store company data on external cloud
servers. This creates two major concerns: the cost of maintaining these systems and
the risk of sensitive employee location data being stored on servers outside the
organization's control.

FieldCheck was created to address these issues directly. It's a mobile application
that uses GPS technology and geofencing to ensure employees can only record their
attendance when they're physically present at authorized work locations. The system
doesn't rely on expensive external services. Instead, it can be hosted locally or on
a private server that the organization controls. This approach significantly reduces
costs while giving companies complete control over their data and privacy.

The system combines two main components: a mobile app for employees and a backend
server for data processing and storage. Together, they create a comprehensive solution
for tracking field-based workers accurately and securely.

Field Area of Project or Study

This project draws from several important areas of technology and business:

- GPS and Geofencing Systems – Using location data to control and verify access
- Workforce Attendance Monitoring – Tools and methods for tracking employee presence
- Location-Based Services – Software that uses geographic positioning for various purposes
- Self-Hosted Mobile Systems – Applications deployed on local infrastructure rather than public clouds
- Mobile Application Development – Building apps using the Flutter framework
- Real-Time Communication Systems – Technologies that enable instant data updates across systems

Problems

General Problem

Organizations with mobile or field-based workers face a significant challenge: how do
you verify that employees are actually where they say they are? Without proper
verification, it's easy for workers to falsify their location, leading to inaccurate
attendance records and wasted company resources.

Specific Problems

1. Fraudulent Attendance Records – Workers can easily claim to be at a job site when
   they're actually elsewhere, making it impossible to trust attendance data

2. Expensive Third-Party Services – Most modern attendance systems require subscriptions
   or API fees, which add up quickly for organizations with many employees

3. Data Privacy Concerns – When companies use cloud-based systems, their sensitive
   employee location data is stored on external servers, raising privacy and security
   questions

4. Poor Performance in Remote Areas – Many systems fail when internet connectivity is
   weak or unavailable, which is exactly when field workers need the system most

5. Disconnected Systems – Attendance tracking and task management often operate
   separately, creating confusion and inefficiency in daily operations

6. Difficult Reporting – Generating reports and exporting data is often a manual,
   time-consuming process that slows down decision-making

Objectives of the Study

General Objective

The main goal of this project is to create a mobile application that tracks employee
attendance accurately and in real-time using GPS and geofencing technology. The system
should work without relying on expensive third-party services, allowing organizations
to maintain complete control over their data while reducing operational costs.

Specific Objectives

1. Design a geofencing system using Flutter Map and GPS that prevents employees from
   checking in unless they're physically at an authorized job location

2. Create a task management feature that lets administrators assign specific tasks to
   employees and track whether those tasks are completed

3. Implement an offline mode that saves attendance data locally when there's no internet
   connection, then automatically uploads it when connectivity returns

4. Build a real-time admin interface within the mobile app that shows live updates of
   employee locations and attendance status without requiring page refreshes

5. Reduce costs by using free mapping tools (OpenStreetMap) instead of expensive
   commercial mapping APIs

6. Add the ability to export attendance and task data as PDF and Excel files for
   reporting and payroll purposes

7. Support both circular and polygon-shaped geofences to accommodate different types
   of job site boundaries

Scope

The system consists of two main parts:

Employee Mobile Application (Flutter): A cross-platform app for Android and iOS with
role-based interfaces for both employees and administrators.

Backend Server: A Node.js and Express server handles all the business logic and data
processing.

The project focuses primarily on GPS-based geofencing. While Bluetooth beacons could
be added in the future for better accuracy in indoor environments, they're not part
of the initial release. The offline functionality is a key feature, allowing the app
to work even when internet connectivity is temporarily lost.

Limitations

Like any project, this system has some constraints:

- GPS Accuracy: The system's accuracy depends on the phone's hardware and environmental
  conditions. Tall buildings or dense forests can interfere with GPS signals

- Initial Login Required: While the app works offline, users need an internet connection
  to log in initially and download their assigned job sites

- Battery Usage: Constantly using GPS drains phone batteries faster than normal apps.
  Users should be aware of this when using the system during long shifts

- Bluetooth Beacons: Adding Bluetooth beacons for better accuracy would require
  additional hardware investment, so this isn't included in the first version

- Complex Calculations: Polygon geofences require more complex calculations, which
  might slow down older phones

Significance of the Study

This project benefits several groups:

Company Management: The system provides a reliable way to verify that employees are
actually at their assigned locations. This prevents fraud and makes payroll processing
easier through automated reporting and data export features.

Field Employees: Workers get a simple, straightforward way to record their work hours.
The offline functionality means they don't have to worry about losing data if they
temporarily lose internet connection. They also have a clear digital record of their
work hours and completed tasks.

The IT Field: This project demonstrates that organizations don't need to pay for
expensive mapping APIs. Open-source solutions like Flutter Map and OpenStreetMap can
provide the same functionality at no cost, making advanced location-based systems
accessible to smaller organizations.

By hosting the system locally, organizations maintain complete control over their data,
reduce ongoing costs, and can customize the system to fit their specific needs.

Definition of Terms

Geofencing: An imaginary boundary drawn around a physical location using GPS coordinates.
When a mobile device enters this boundary, the system can trigger specific actions, like
allowing an attendance check-in.

JWT (JSON Web Token): A secure way to verify that a user is who they claim to be. It's
like a digital ID card that proves a user is authenticated without storing passwords.

MongoDB: A database system that stores information in a flexible, document-based format,
similar to how information might be organized in JSON files.

Socket.io: A technology that allows real-time communication between a server and mobile
or web clients, enabling instant updates without requiring users to refresh their screens.

Offline Synchronization: The process of saving data locally on a device when internet
is unavailable, then automatically uploading that data to the central server once
connectivity returns.

Flutter: An open-source framework created by Google for building mobile apps that work
on both Android and iOS from a single codebase.

Flutter Map: An open-source package for Flutter that provides interactive mapping
capabilities using free OpenStreetMap data, eliminating the need for paid mapping APIs.

RBAC (Role-Based Access Control): A security approach that determines what different
types of users can do in the system. For example, administrators might have access to
all features, while regular employees only see their own data.

CHAPTER 2: REVIEW OF RELATED LITERATURE
================================================================================

The way organizations track employee attendance has evolved significantly over the
past few decades. What started as simple paper logs has gradually transformed into
digital systems powered by mobile technology and location data. This evolution has
been driven by the limitations of older methods—they're easy to fake, expensive to
maintain, and don't work well in field environments.

Recent research shows that GPS and geofencing technology have become the standard
approach for modern attendance systems. Researchers like Kurnia and colleagues (2025)
developed mobile apps that verify workers are at specific locations before recording
attendance. Similarly, Shinde's team (2022) created systems that use geofencing to
improve accuracy in monitoring field workers. These solutions have proven effective
at preventing fraudulent check-ins and reducing the need for expensive hardware-based
biometric systems.

Many researchers have gone further by combining geofencing with biometric technology.
For example, Aigbokhan and colleagues (2024) created a system that uses both GPS and
facial recognition to verify both location and identity. This two-layer approach is
more secure because it prevents someone from checking in on behalf of another worker.
Other researchers, like Pawar's team (2024), added real-time face detection to prevent
spoofing attacks where someone uses a photo instead of appearing in person.

Beyond just attendance tracking, researchers have explored how geofencing can be part
of a larger workforce management system. GeoWorkTrack (Tile et al., 2024) and similar
systems extend geofencing to include employee tracking, task assignment, and productivity
monitoring. This shows that geofencing isn't just useful for attendance—it can be part
of a comprehensive management solution.

The technology has been applied in many different settings. Universities use it to track
student attendance (Mohammed, 2021; Eweoya et al., 2025), companies use it for employee
monitoring, and some organizations even use it for child safety monitoring (Agajo et al.,
2023). This wide range of applications demonstrates how versatile geofencing technology
really is.

However, researchers have also identified challenges that need to be addressed. Battery
drain and connectivity issues can limit GPS-based systems (Harshada et al., 2023).
Rexline and Nazrin (2024) suggested using lighter-weight approaches to reduce power
consumption, while Fatkharrofiqi's team (2020) showed how to optimize location-based
services for Android to minimize system overhead.

When researchers reviewed the overall state of attendance technology, they identified
clear trends. Ali and colleagues (2022) found that geolocation, biometrics, and cloud
integration are the leading approaches. Abdalkarim and Sakarya (2022) confirmed that
the industry is moving away from hardware-based solutions toward mobile-first systems
that are more flexible and easier to use.

Security and fraud prevention are also major concerns in the literature. Olajide (2024)
explored multiple ways to prevent fraud, including dynamic QR codes, geofencing, and
device verification. Other researchers emphasize the importance of encryption, proper
access controls, and secure system design to protect employee privacy while preventing
fraud (Gupta, 2022; Afolab, 2021).

In summary, the research community has clearly established that geofencing-based
attendance systems are the way forward. The evidence shows they're more secure than
traditional methods, they can be combined with other technologies like biometrics for
extra security, and they work across many different industries and settings. The main
challenges are managing battery usage, ensuring systems work in areas with poor
connectivity, and protecting employee privacy. FieldCheck addresses these challenges
by combining proven geofencing technology with offline functionality and a self-hosted
architecture that gives organizations complete control over their data.

3.1 Project Design

Research Design

This project uses a Developmental Research Design combined with an Agile development
methodology. Developmental Research Design means we're not just studying something that
already exists—we're actually building and testing a new technology solution. This
approach makes sense because our main goal is to create a working system and then
evaluate how well it performs in real-world conditions.

The Agile methodology guides how we build the system. Rather than planning everything
upfront and building it all at once (Waterfall), we work in short cycles, building
features incrementally, testing them, and improving based on feedback. This allows us
to adapt quickly if we discover new requirements or challenges during development.

Overview

FieldCheck is built as a two-part system. The first part is a mobile app for employees,
built using Flutter so it works on both Android and iOS phones. The second part is a
backend server for data processing and storage.

The mobile app lets employees check in and out at job sites using GPS verification.
It also handles task management, stores data locally when there's no internet, and
can generate reports.

Core Design Principles

We built FieldCheck around several key ideas:

Accuracy – The system uses GPS to pinpoint employee locations. It can handle both
simple circular boundaries and complex polygon-shaped boundaries. In the future,
Bluetooth beacons could be added for even better accuracy indoors.

Privacy and Control – The system doesn't rely on any external services. Everything
is hosted locally or on a private server that the organization controls. This means
the company owns all its data.

Reliability – The system works even when the internet is down. Data is stored locally
and automatically syncs when connectivity returns. The system is designed to handle
errors gracefully.

Cost Efficiency – We use free, open-source mapping tools instead of expensive
commercial APIs. This dramatically reduces costs.

Scalability – The system is built in a modular way, so it can handle more employees,
more job sites, and more concurrent users as the organization grows.

Usability – Both employees and administrators should be able to use the system without
extensive training. The interfaces are designed to be intuitive and straightforward.

System Architecture

The system has several layers:

Frontend: Flutter mobile app with role-based interfaces for both employees and
administrators.

Backend: A Node.js and Express server handles all the business logic and data processing.

Mapping: We use Flutter Map with OpenStreetMap data for creating and displaying
geofences. This replaces the need for expensive commercial mapping APIs.

Data Storage: All data is stored on a local or private server, not on external clouds.

Security: The system uses role-based access control, encrypted data transmission,
JWT tokens for authentication, and optional two-factor authentication.

Real-Time Updates: Socket.io enables instant updates across the system without
requiring users to refresh their screens.

3.2 System Development Life Cycle (Agile Methodology)

We're using an Agile approach to development, which means we work in short cycles
(sprints) and continuously improve the system based on feedback. Unlike Waterfall
methodology where all planning happens first and then execution, Agile allows us to
adapt and refine requirements as we learn more about the system.

Our Agile process follows these phases:

Planning & Requirements: We start by understanding what the system needs to do,
identifying who will use it, and mapping out features. We prioritize features based
on business value and technical feasibility.

Design: We create diagrams showing how data flows through the system, design the
user interfaces, and plan out the algorithms for geofencing. Design decisions are
revisited in each sprint as needed.

Implementation: We build the mobile app and backend server in parallel, writing
clean, modular code that's easy to maintain. Features are developed incrementally
in short sprints (typically 1-2 weeks).

Testing: We test individual components (unit testing), test how they work together
(integration testing), and test the system in real field conditions (user acceptance
testing) to ensure GPS accuracy and offline sync work properly.

Deployment: We set up the server and test the system with real users. Feedback from
users informs the next sprint.

Maintenance & Feedback: We continue to update the system with security patches,
performance improvements, and new features based on user feedback. This cycle repeats
continuously.

How This System Works

Imagine an employee arrives at a construction site. They open the FieldCheck app on
their phone and tap the check-in button. Behind the scenes, the app reads their GPS
location and compares it to the job site boundaries that the administrator set up.
If the employee is within the boundary, the check-in is recorded. If they're outside
the boundary, the app shows an error message and won't let them check in.

The system supports two types of boundaries. Simple circular boundaries are defined
by a center point and a radius—like a circle around a specific location. More complex
polygon boundaries are defined by multiple points that form an irregular shape—useful
for job sites that aren't circular.

If the employee is offline when they try to check in, the app saves the check-in
locally on their phone. When they get internet connectivity again, the app automatically
sends all the saved data to the server. This ensures no attendance records are lost,
even in remote areas with spotty connectivity.

On the administrative side, managers use the admin interface within the mobile app to
set up job sites on an interactive map. They can draw circular or polygon boundaries,
assign employees to specific sites, and view attendance records in real-time. They can
also generate reports and export data as PDF or Excel files for payroll and auditing
purposes.

All the data is stored on the organization's own server, not on an external cloud.
This gives the organization complete control over sensitive employee location data
and eliminates the need to pay for expensive third-party services.

Process Model

The FieldCheck system works through a series of steps:

First, an administrator logs into the mobile app using their admin credentials. The
app recognizes their admin role and displays the admin interface. They can create
geofenced job sites on an interactive map. They can define circular boundaries by
specifying a center point and radius, or polygon boundaries by plotting multiple
points. Once the boundaries are set, they assign employees to their designated work
sites.

When an employee arrives at a job site, they open the mobile app and tap check-in.
The app reads their GPS location and compares it to the geofence boundaries. For
circular geofences, the app uses a mathematical formula called the Haversine formula
to calculate the distance between the employee and the center of the geofence. For
polygon geofences, the app uses a technique called ray casting to determine if the
employee is inside the boundary.

If the employee is within the geofence, the check-in is recorded with the time and
location. If they're outside, the app rejects the check-in and tells the user they
need to be within the job site boundary.

Throughout the day, employees can check out using the same process. Each attendance
record is stored locally on the phone. If the internet is down, the records stay on
the phone. When connectivity returns, the app automatically syncs all the records
with the backend server.

The backend server processes all the attendance data and stores it in the organization's
database. Administrators can then view this data through the admin interface within
the mobile app, filter it by employee or date, and export it for reporting and payroll
processing.

This cycle repeats every day, creating an automated system that ensures attendance
is only recorded when employees are actually at their assigned locations.

3.6 Algorithms and Technical Details

Circular Geofence Validation

For circular geofences, the system uses the Haversine formula to calculate the
distance between the employee's GPS coordinates and the geofence center:

The formula calculates the great-circle distance between two points on a sphere given
their latitudes and longitudes. If this distance is less than or equal to the geofence
radius, the employee is within the geofence.

Polygon Geofence Validation

For polygon geofences, the system uses the ray casting algorithm. This algorithm works
by drawing an imaginary ray from the employee's location to infinity. If the ray crosses
the polygon boundary an odd number of times, the employee is inside the polygon. If it
crosses an even number of times, the employee is outside.

This algorithm is efficient and works for any polygon shape, making it ideal for
irregular job site boundaries.

Offline Synchronization

When an employee is offline, the app stores attendance records locally using SQLite
or Hive (local storage databases). When connectivity returns, the app:

1. Detects the restored connection
2. Retrieves all stored records from local storage
3. Sends them to the backend server
4. Waits for confirmation from the server
5. Deletes the records from local storage once confirmed

This ensures no data is lost and the system stays in sync.

Real-Time Updates

Socket.io maintains a persistent connection between the app and the server.
When something changes (like a new attendance record or task assignment), the server
broadcasts an update to all connected administrators. This means administrators see
updates instantly without refreshing their screen.

3.7 Security Implementation

Authentication: The system uses JWT tokens for authentication. When a user logs in,
the server creates a token that proves the user's identity. This token is sent with
every request to the server.

Authorization: The system implements role-based access control. Administrators can
access all features, while employees can only see their own data and assigned tasks.

Data Encryption: All data transmitted between the app and server is encrypted using
HTTPS/TLS. This prevents anyone from intercepting sensitive information.

Password Security: User passwords are hashed using bcryptjs before being stored in
the database. This means even if someone gains access to the database, they can't
read the passwords.

4.3 Key Features

Geofencing Algorithms

Circular Geofencing: Uses the Haversine formula to calculate distance between employee
location and geofence center. If distance ≤ radius, employee is within geofence.

Polygon Geofencing: Uses ray casting algorithm to determine if employee is within
polygon boundary. This supports irregular-shaped job sites.

Offline Synchronization

The system stores attendance data locally when offline. When connectivity returns:
1. App detects connection
2. Retrieves stored records
3. Sends to backend server
4. Deletes from local storage after confirmation

This ensures no data loss in areas with poor connectivity.

Real-Time Updates

Socket.io maintains persistent connections between clients and server. When data
changes, the server broadcasts updates instantly to all connected clients. This means:

- Administrators see new attendance records immediately
- Task assignments appear instantly on employee devices
- Geofence updates are reflected in real-time

4.5 Challenges and Solutions

Challenge 1: GPS Accuracy in Urban Areas

Problem: Tall buildings and dense vegetation interfere with GPS signals, reducing
accuracy.

Solution: Implemented GPS spike filtering to remove anomalous readings. Added support
for optional Bluetooth beacons for enhanced accuracy in problematic areas.

Challenge 2: Battery Consumption

Problem: Continuous GPS usage drains device batteries quickly.

Solution: Implemented intelligent GPS sampling—the app only reads GPS at intervals
rather than continuously. It also allows users to disable GPS when not needed.

Challenge 3: Offline Data Synchronization

Problem: Ensuring data consistency when syncing large amounts of offline data.

Solution: Implemented a queuing system that syncs data in batches. The system tracks
which records have been synced to prevent duplicates.

Challenge 4: Polygon Geofence Performance

Problem: Ray casting algorithm can be slow for very complex polygons on older devices.

Solution: Implemented polygon simplification to reduce the number of points in complex
polygons while maintaining accuracy. This significantly improved performance.
